## 场景判断

| 场景 | 做法 |
|------|------|
| 用户给了简历模板（空白 Word） | 解包 → 理解结构 → 填入内容 → 打包 |
| 用户给了旧简历（有内容的 Word/PDF） | 提取内容 → 编辑优化 → 回填到原文件 |
| 用户什么都没给 | 通过对话收集信息 → 生成新文件 |

## 读取用户文件

### Word 文件

两种读取方式，用途不同：

**方式一：提取文本（用于了解内容）**

```bash
uv run scripts/extract_docx.py resume.docx
```

输出带结构标记的 Markdown 文本：标题用 `##`，列表用 `-`，加粗用 `**`。适合快速了解简历内容和结构。

```bash
# JSON 格式输出（每个元素标记 type、style、level 等）
uv run scripts/extract_docx.py resume.docx --format structured
```

**方式二：解包为 XML（用于编辑回填）**

```bash
uv run scripts/docx_edit.py unpack resume.docx unpacked/
```

解包后在 `unpacked/word/document.xml` 中可以看到完整的 XML 结构。直接在这里修改文本，然后打包回去，格式完全保留。

**通常两种都要用：** 先用 extract 了解内容，再用 unpack 准备编辑。

### PDF 文件

```bash
uv run scripts/extract_pdf.py resume.pdf
```

PDF 只能读取不能回填。如果用户只有 PDF 而需要可编辑的文件，选择之一：
- 让用户找到原始 Word 文件
- 从提取的内容出发，生成新的 Word 文件
- 用 pandoc 尝试转换（效果不一定理想）

### 含修订标记的 Word 文件

```bash
pandoc --track-changes=all resume.docx -o resume.md
```

## 填写模板

用户给了一个简历模板（空白或半空的 Word 文件），需要填入信息。

### 流程

**1. 提取模板结构**

```bash
uv run scripts/extract_docx.py template.docx
```

看清楚模板有哪些板块、占位符文本是什么（比如"在此填写姓名"、"[公司名称]"）。

**2. 解包模板**

```bash
uv run scripts/docx_edit.py unpack template.docx unpacked/
```

**3. 了解 XML 中的占位符**

读取 `unpacked/word/document.xml`，找到占位符文本在 XML 中的位置。比如：

```xml
<w:r>
  <w:rPr><w:b/><w:sz w:val="28"/></w:rPr>
  <w:t>在此填写姓名</w:t>
</w:r>
```

**4. 替换占位符**

用编辑工具将 `<w:t>` 中的占位符文本替换为真实内容。保留 `<w:rPr>` 不动——这就是保持格式的关键。

```xml
<w:r>
  <w:rPr><w:b/><w:sz w:val="28"/></w:rPr>
  <w:t>张三</w:t>
</w:r>
```

**5. 打包**

```bash
uv run scripts/docx_edit.py pack unpacked/ filled_resume.docx
```

### 注意事项

- 模板中的占位符可能被拆分到多个 `<w:r>` 中（Word 经常这样做），需要手动合并相邻 run 再替换
- 如果占位符跨多个 run，需要将它们合并为一个 run 再替换
- 不要改动 `<w:rPr>`（格式属性）、`<w:pPr>`（段落属性）、`<w:sectPr>`（页面设置）等非文本内容

## 从零构建

用户没有模板也没有旧简历时，通过对话收集信息。

### 收集顺序

不要一次问完。分步来：

**第一步：基础信息**
- 姓名、联系方式
- 当前/最近的工作和职位
- 目标岗位（如果有）

**第二步：工作经历**
- 从最近的工作开始，逐个展开
- 每个职位问 3-5 个核心成就
- 如果用户说不出来，切换到辅导模式（参见 `references/简历辅导.md`）

**第三步：补充板块**
- 教育背景、技能、项目、证书等

### 对话开场

> "先告诉我你最近的工作——在哪家公司、做什么岗位、干了多久？"

不要从"你想做什么格式的简历"开始。先收集内容，格式最后再说。

### 收集完成后

内容收集完毕后，需要生成文件。此时有两条路：

1. **用 docx-js 生成新 Word 文件** — 需要写 Node.js 脚本，格式规范见 `references/格式检查.md`
2. **用 Typst 生成 PDF** — 见 `references/格式检查.md` 的 PDF 生成章节

## 内容组织原则

### 成就描述公式

```
[强动作动词] + [做了什么] + [技术/方法] + [量化成果]
```

**好的例子：**
- 主导搭建实时风控系统，基于 Flink 处理引擎，日均处理 5000 万笔交易，误报率降低 40%
- 设计并落地微服务拆分方案，将单体应用拆为 12 个服务，部署频率从月度提升至日级

**差的例子：**
- 负责后端开发
- 参与了数据库优化

### 篇幅控制

| 经验水平 | 目标页数 | 每个职位描述条数 |
|----------|---------|-----------------|
| 0-3 年 | 1 页 | 3-4 条 |
| 3-8 年 | 1-2 页 | 4-5 条（近期）/ 2-3 条（早期） |
| 8 年以上 | 2 页 | 4-6 条（近期）/ 1-2 条（早期） |

### 摘要公式

`[职级/角色] + [工作年限] + [核心能力] + [行业经验] + [关键价值]`

> 拥有 8 年后端开发经验的高级工程师，专注于分布式系统和微服务架构。在金融科技和电商领域主导过多个核心系统的设计与落地，擅长高并发场景下的性能优化。
